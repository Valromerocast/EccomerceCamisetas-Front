1. Resumen ejecutivo

Mundialista Store es una SPA de e-commerce full stack. Los compradores exploran y filtran camisetas, eligen variantes por talle, administran favoritos y carrito, y confirman pedidos. Los administradores gestionan productos, variantes, stock, usuarios y estados de pedidos.

SPA

Cliente

React renderiza la interfaz sin recargar el documento completo. React Router resuelve las vistas.

REST

Comunicación

El frontend usa fetch y JSON contra endpoints HTTP del backend.

3 capas

Servidor

Controller recibe HTTP, Service aplica negocio y Repository persiste.

JWT

Seguridad

Sesiones stateless y autorización por roles USER y ADMIN.

Pitch de 30 segundos: “Desarrollamos un e-commerce de camisetas conectado de punta a punta. React y Redux Toolkit manejan la experiencia y el estado global; Spring Boot expone una API REST organizada en tres capas; JPA persiste en PostgreSQL; y Spring Security protege las operaciones con JWT y roles. El checkout se resuelve transaccionalmente: valida stock, crea el pedido, descuenta las variantes y vacía el carrito.”

2. Guion recomendado para la demo

  1. Inicio y catálogo: explicar que es público, mostrar búsqueda, filtros por país/tipo/género/talle/precio y ordenamiento.
  2. Detalle: seleccionar talle; remarcar que el stock pertenece a una variante, no a la camiseta general.
  3. Autenticación: iniciar sesión como comprador. Explicar JWT y restauración de sesión.
  4. Favorito y carrito: agregar una camiseta, modificar cantidad y señalar que todo persiste en backend.
  5. Checkout: confirmar. Contar la transacción: bloqueo/lectura de stock, validación, pedido, descuento y vaciado.
  6. Perfil: mostrar el historial real del usuario.
  7. Panel admin: entrar con rol ADMIN; mostrar inventario, edición de variantes y estados de pedidos.
  8. Swagger: cerrar mostrando la API y una respuesta JSON.
No arriesgar datos compartidos durante el oral. Preparar previamente un producto y un carrito de prueba. Una compra descuenta stock y crea un pedido permanente; una baja de camiseta es lógica pero también persiste.

¿Monolito o microservicios?

Es un monolito modular: una sola aplicación Spring Boot y un solo despliegue de backend, pero separada internamente por responsabilidades. Para el tamaño académico reduce complejidad operativa, latencia entre servicios y problemas de consistencia distribuida.

Separación de responsabilidades

Controller

Traduce HTTP a llamadas de aplicación: rutas, métodos, request DTO y status.

Service

Concentra reglas, validaciones, transacciones y conversión a responses.

Repository

Abstrae consultas y persistencia mediante Spring Data JPA.

DTO

Define el contrato externo y evita exponer directamente las entidades.

¿Por qué Redux Toolkit?

Hay datos compartidos por muchas vistas y componentes: usuario, productos, carrito, favoritos y pedidos. Redux ofrece una única fuente de verdad, actualizaciones predecibles y herramientas para asincronismo. createAsyncThunk modela los estados pending / fulfilled / rejected; los slices actualizan el estado y los selectores derivan datos.

Estado global vs local: los datos de negocio compartidos van a Redux. Campos de formulario, modales o selección visual temporal quedan en useState.

Selectores memoizados

createSelector enriquece carrito y pedidos con el catálogo actual. La memoización evita recalcular si sus entradas no cambiaron.

Adaptación de datos

services/api.js centraliza URL, token, errores y mapeos. Por ejemplo, convierte PENDIENTE del backend en “Procesando” para la interfaz y agrupa variantes para producir stock por talle.

Manejo centralizado de errores

@RestControllerAdvice transforma excepciones en respuestas uniformes: 400 para validación/negocio, 401 para falta de autenticación, 403 para permisos, 404 para recursos inexistentes, 409 para conflictos y 500 para errores inesperados.

Ventaja: los controllers no repiten bloques try/catch, y el frontend siempre puede leer un contrato de error predecible.

La decisión clave: camiseta vs variante

Camiseta guarda información común: nombre, descripción, precio, imagen, país, género y tipo. CamisetaTalle representa una variante vendible con talle, color, SKU y stock. Así no se duplica la ficha completa por cada talle y se controla stock de forma correcta.

¿Por qué BigDecimal para dinero? Porque float y double usan representación binaria y pueden introducir errores de precisión. BigDecimal ofrece cálculo decimal exacto.

Detalles que preservan historia

DetallePedido guarda cantidad, precio unitario y subtotal al comprar. Si el precio de la camiseta cambia después, el pedido conserva el precio histórico.

LAZY, cascade y orphanRemoval

  • FetchType.LAZY evita traer relaciones hasta necesitarlas.
  • cascade = ALL propaga operaciones del agregado padre a sus hijos.
  • orphanRemoval = true elimina hijos que dejan de pertenecer a la colección.

401 vs 403

401 Unauthorized: no hay credenciales válidas. 403 Forbidden: sí está autenticado, pero el rol no tiene permiso. Por ejemplo, un ADMIN autenticado recibe 403 al intentar usar el carrito.

Punto honesto: guardar JWT en localStorage simplifica la SPA, pero un XSS podría leerlo. En producción convendría evaluar cookies HttpOnly + Secure + SameSite, CSP y una estrategia de refresh tokens.

8. Flujo crítico: checkout

  1. El usuario envía POST /api/pedidos.
  2. El backend identifica al usuario desde el contexto autenticado.
  3. Busca usuario y carrito; rechaza un carrito vacío.
  4. Por cada ítem obtiene la variante con una consulta for update.
  5. Valida camiseta activa y stock suficiente.
  6. Copia precio/cantidad/subtotal a un detalle histórico.
  7. Descuenta stock y calcula el total con BigDecimal.
  8. Guarda pedido y detalles, y vacía el carrito.
¿Por qué @Transactional? Todo debe ocurrir como una unidad atómica. Si falla una variante, se revierte la operación completa: no debería quedar un pedido parcial ni stock descontado a medias.

Concurrencia

La consulta findByIdForUpdate indica un bloqueo pesimista de la variante durante la transacción. Evita que dos compras simultáneas lean el mismo último stock y ambas lo vendan.

Cancelación

Solo un pedido PENDIENTE puede cancelarse por el usuario. Cancelar repone stock. La operación es idempotente para un pedido ya cancelado: no repone dos veces. Una vez cancelado, no puede pasar a otro estado.

10. Teoría aplicada al proyecto

API vs Web Service
Una API es un contrato para que dos piezas de software interactúen. Un web service es una API accesible mediante tecnologías web. Este proyecto expone una API web REST sobre HTTP y JSON.
REST: ¿qué significa?
Es un estilo arquitectónico orientado a recursos, identificados por URI y manipulados con una interfaz uniforme HTTP. El servidor es stateless y las representaciones viajan principalmente como JSON. No basta con “usar HTTP”: importan recursos, semántica de métodos y códigos.
SPA vs MPA
En una SPA se carga un documento base y JavaScript cambia vistas en el cliente; React Router evita recargas completas. En una MPA, cada navegación suele pedir un nuevo documento HTML al servidor.
DOM y Virtual DOM
El DOM es la representación en memoria del documento. React crea una representación virtual, compara cambios y actualiza solo las partes necesarias del DOM real.
Props vs state
Las props son entradas que un componente recibe. El state es información mutable controlada por el componente o por un store externo. Las props no deberían modificarse desde el hijo.
useEffect
Ejecuta efectos secundarios luego del render, como cargar datos o restaurar sesión. Su array de dependencias determina cuándo se vuelve a ejecutar.
Promesas y async/await
fetch devuelve una Promise. await pausa esa función async hasta que se resuelve, sin bloquear el hilo del navegador. Los errores se capturan y terminan en el estado rejected del thunk.
ORM y JPA
ORM mapea objetos a tablas. JPA es la especificación Java; Hibernate es la implementación usada por Spring Boot. Las anotaciones describen entidades, claves y relaciones.
Inyección de dependencias
Spring construye services, repositories y controllers y les entrega sus dependencias por constructor. Reduce acoplamiento, mejora testabilidad y centraliza el ciclo de vida.
CORS
Es una política del navegador para requests entre orígenes distintos. Backend 8080 y frontend 5173 son orígenes diferentes; la configuración permite localhost, métodos y headers necesarios.
CSRF deshabilitado
El backend usa autenticación stateless mediante un token enviado explícitamente en un header, no una sesión tradicional basada en cookie. Por eso se deshabilita CSRF en esta configuración. Si se migrara el JWT a cookies, habría que reevaluarlo.
DTO vs entidad
La entidad representa persistencia; el DTO representa el contrato HTTP. Separarlos evita filtrar password o relaciones internas, previene ciclos de serialización y permite evolucionar API y base de forma independiente.

11. Posibles preguntas de la profesora

Usá el buscador de arriba. Intentá contestar antes de abrir cada respuesta.

¿Dónde está la regla de negocio: controller, service o repository?
Principalmente en Service. El controller maneja HTTP y el repository acceso a datos. Ejemplo: validar stock y reponerlo al cancelar están en PedidoService.
¿Por qué no validar el stock solo en React?
Porque el cliente puede manipularse y su información puede quedar desactualizada. La validación autoritativa debe estar en backend y repetirse al confirmar la compra.
¿Cuál es la fuente de verdad del carrito?
El backend y PostgreSQL. Redux conserva una copia para renderizar, pero cada mutación espera la respuesta real de la API.
¿Qué persiste localStorage?
Solo el JWT. Productos, usuarios, favoritos, carrito, pedidos y stock persisten en backend.
¿Por qué Redux si existe Context?
Context sirve para dependencias globales simples, pero Redux Toolkit aporta slices, thunks, DevTools, selectores memoizados y un flujo más explícito para varios dominios asíncronos.
¿Qué es un thunk?
Una acción asíncrona que puede ejecutar lógica, llamar a la API y luego despachar resultados. createAsyncThunk genera pending, fulfilled y rejected.
¿Qué ocurre al recargar el navegador?
App carga productos, busca el JWT y consulta /api/usuarios/me. Si es válido restaura usuario y carga sus datos; si falla elimina el token.
¿JWT cifra la información?
Normalmente no: la firma garantiza integridad y autenticidad, pero el payload Base64URL puede leerse. Nunca se deben poner secretos en él.
¿Cómo se guardan las contraseñas?
Se hashean con BCrypt mediante PasswordEncoder. No se guardan en texto plano ni se pueden “descifrar”; se compara el hash.
¿Qué evita que un USER llame endpoints de ADMIN manualmente?
Spring Security valida el rol en el backend. Ocultar botones en React mejora UX, pero no es seguridad.
¿Por qué el ADMIN no puede comprar?
Es una regla de roles del proyecto: carrito y favoritos requieren ROLE_USER. Separar responsabilidades reduce acciones incoherentes con la cuenta administrativa.
¿Qué hace @Transactional?
Delimita una transacción de base de datos. Todas las operaciones se confirman juntas o se revierten ante una excepción. Es esencial en checkout y cancelación.
¿Qué problema de concurrencia puede ocurrir con el último producto?
Dos transacciones podrían leer stock 1 y vender ambas. El repository obtiene la variante para actualización, bloqueando la fila durante la transacción.
¿Por qué guardar precio en DetallePedido?
Para preservar la fotografía histórica de la compra. Si el catálogo cambia, el total anterior no debe cambiar.
¿DELETE de camiseta es físico?
La implementación usa baja lógica: cambia activo y el catálogo filtra inactivas. Conserva integridad e historia vinculada.
¿Qué significa que cancelar sea idempotente?
Repetir la misma cancelación no produce un efecto adicional: si ya está cancelado, no vuelve a sumar stock.
¿Qué son cascade y orphanRemoval?
Cascade propaga operaciones del padre a hijos. Orphan removal elimina de base un hijo removido de la colección administrada.
¿Por qué relaciones LAZY?
Para no cargar grafos completos innecesariamente. Mejora rendimiento, aunque exige acceder dentro de una sesión/transacción o mapear a DTO.
¿Qué es el problema N+1?
Una consulta carga N entidades y luego se ejecuta una consulta adicional por cada relación. Puede resolverse con fetch joins, entity graphs o consultas específicas.
¿PUT y PATCH son iguales?
No. PUT suele reemplazar el recurso completo o su representación editable; PATCH modifica parcialmente. El proyecto usa PATCH para cantidad, stock, contraseña y estado.
¿Qué códigos de error usa la API?
400 negocio/validación, 401 credenciales, 403 permisos, 404 inexistente, 409 integridad y 500 inesperado.
¿Por qué existe api.js?
Centraliza la base URL, headers, JWT, parsing de errores y adaptación backend↔frontend. Evita duplicación y desacopla componentes del contrato crudo.
¿Qué prueba un test unitario con Mockito?
La lógica de una unidad aislada, reemplazando repositorios por mocks. Los tests verifican, por ejemplo, reposición de stock, no duplicar reposición y checkout de la última unidad.
¿Es suficiente el testing actual?
No. Hay tests unitarios valiosos, pero faltan más tests de carrito, auth, controllers, integración con base y frontend/E2E.
¿Qué cambiarías para producción?
Migraciones Flyway/Liquibase, secretos fuera de la entrega, cookies HttpOnly o estrategia segura de tokens, validación de estados permitidos, observabilidad, paginación, tests integrales y despliegue con HTTPS.

12. Límites reales y mejoras

Reconocer límites con criterio suele valer más que defender algo indefendible.

Checkout incompleto

Dirección y método de pago no persisten en Pedido; solo sirven para la confirmación inmediata del frontend.

Estados libres

El estado es String. Sería más seguro usar enum y transiciones explícitas.

DDL automático

ddl-auto=update sirve para desarrollo; producción debería usar migraciones versionadas.

Token

localStorage es práctico, pero tiene riesgo ante XSS.

Imágenes

Se persiste una URL pública, no un servicio propio de almacenamiento o múltiples imágenes.

Testing

La cobertura actual se concentra en pedidos, favoritos y precios; faltan suites de integración y E2E.
